# 相册模块（照片模块）PRD（重构版 v2）

## 1. 文档说明

### 1.1 文档目标
本 PRD 用于定义当前阶段「相册/照片模块」的最终产品规则，覆盖：

- 信息架构与导航
- 照片页（全局媒体流）
- 相册页与帖子体系
- 帖子详情与查看态
- 系统媒体工具区
- 评论体系
- 原图加载
- 删除与回收站
- 通知与设置
- 导出与保存

本版以你当前已经确认的产品方向为准，不再沿用旧版里“帖子评论与媒体评论混流、照片页默认帖子视角、系统媒体入口较弱、查看态评论弹层过于遮挡”的旧口径。

---

### 1.2 产品定位
该模块是一个**两人协作使用**的照片系统，强调：

- 浏览顺手，整体体验接近系统相册；
- 进入 app 后先看媒体，而不是先看复杂的内容结构；
- 帖子继续承担内容容器角色，但不过度压过媒体本身；
- 评论分层明确，避免“对整体说的话”和“对某张图说的话”混在一起；
- 删除语义清楚，恢复路径明确；
- 保持沉浸感，不让查看态被厚重操作层打断。

---

### 1.3 本版关键决策
本版统一采用以下规则：

1. **全局底部导航固定为：`主页 / 照片 / 生活（暂定）`**
2. **进入“照片”后，顶部二级导航为：`照片 / 相册 / 回收站`**
3. **通知入口位于照片模块顶部区域右侧铃铛**
4. **系统媒体为独立工具区，不混入 app 主内容流**
5. **系统媒体入口需要更显眼、更方便，不再只作为弱图标入口**
6. **照片页只展示 app 内容体系内的全局媒体流**
7. **保留“帖子”概念**
   - 相册：帖子集合的组织层
   - 帖子：一组媒体的内容容器
   - 媒体：最小浏览与评论单位
8. **一个媒体可以属于多个帖子**
9. **评论拆分为两类，且不再混流**
   - 帖子评论：只归帖子
   - 媒体评论：只归媒体
10. **查看态中的媒体评论改为沉浸式双层结构**
    - 默认不直接弹出 2/3 屏评论区
    - 先通过画布上的评论气泡 + 半透明评论预览层展示
    - 需要深入查看或输入时，再展开完整评论区
11. **帖子内查看态与媒体流查看态尽量统一**
    - 仅在帖子内查看态额外保留分段小白条
12. **帖子详情页支持“加载全帖原图”**
13. **照片页中删除媒体 = 系统删**
14. **帖子内删除媒体时，提供“目录删 / 系统删”两种语义**
15. **删除整个帖子时，也提供“仅删帖子 / 删除帖子并系统删其中媒体”两种语义**
16. **评论不做回复功能，不做楼中楼，不做引用回复**
17. **数据模型与接口契约、验收口径本版不写，另做独立文档**

---

## 2. 背景、目标与范围

### 2.1 背景
当前产品面向两人长期使用的照片记录与协作场景。

目标不是把 app 做成一个复杂的内容平台，而是做成：

- 日常可快速浏览的媒体空间；
- 能沉淀成片段、记录和帖子；
- 能比系统相册更清晰地承载评论、回看、删除恢复和长期整理。

---

### 2.2 核心目标

#### 1）照片页足够自然
- 用户进入照片模块后先看到的是媒体流；
- 浏览逻辑接近系统相册；
- 全局媒体流保持轻量，不被过多关系信息打断。

#### 2）帖子继续承担内容组织作用
- 一组相关媒体可以组织为一个帖子；
- 帖子承载标题、简介、帖子时间和帖子评论；
- 用户既能看全局媒体，也能回到具体帖子上下文。

#### 3）评论语义清晰
- 对整组内容说的话写在帖子评论里；
- 对具体某张图、某个视频说的话写在媒体评论里；
- 两类评论分别展示，不再混在一个总评论区里。

#### 4）查看态保持沉浸感
- 不让媒体评论默认用厚重底部弹层直接盖住图片；
- 先轻量预览，再按需展开完整评论区；
- 原图、时间、所属帖子等操作既可达，又不过分打断观看。

#### 5）删除语义明确
- 区分“从当前帖子移除”和“全局删除媒体”；
- 区分“删除帖子本体”和“连其中媒体一起系统删除”；
- 回收站能够支撑这些不同语义的恢复。

---

### 2.3 本 PRD 覆盖范围
本 PRD 覆盖：

- 照片二级页
- 相册二级页
- 回收站二级页
- 系统媒体工具区
- 帖子（Post）
- 媒体（Media）
- 帖子评论（Post Comment）
- 媒体评论（Media Comment）
- 删除与回收站
- 通知与设置
- 导出与保存

---

### 2.4 明确不做 / 非目标
本阶段明确不做：

- 文件模块
- 内置相机
- 评论回复
- 楼中楼
- 引用评论
- 评论搜索
- 子相册
- 把 app 内容直接塞入系统选图器成为自定义源
- 在系统媒体区做评论
- 系统媒体区跳转“已上传对应帖子”
- 复杂站外分享链路
- 数据模型与接口契约正文
- 验收口径正文

---

### 2.5 已定实现口径
- 界面语言：中文
- 并发冲突处理：最后写入胜出（LWW）
- 评论默认加载：10 条
- 展开更多评论入口：列表底部
- 系统媒体删除：单张与批量都只弹一次系统确认
- 系统媒体权限策略：尽力引导用户开启所需权限
- 长图默认不在详情页完全展开
- 原图默认不预取，按需加载

## 3. 核心对象与术语

### 3.1 相册（Album）
相册是帖子集合的组织层。

它负责：

- 主题分类
- 归档管理
- 浏览某一集合下的帖子

相册不是默认浏览入口，默认浏览入口仍是“照片”页。

---

### 3.2 帖子（Post）
帖子是 app 内容体系中的**内容容器**。

一个帖子可以承载：

- 多张图片 / 视频 / GIF / 长图等媒体
- 标题
- 简介
- 帖子时间
- 帖子评论
- 所属相册关系

帖子是“内容组织单位”，但不是照片模块的默认第一视角。

---

### 3.3 媒体（Media）
媒体是最小浏览与评论单位。

每个媒体具备：

- 媒体自身时间
- 缩略图 / 详情预览 / 原图
- 媒体评论
- 可被多个帖子引用

照片页按媒体浏览；帖子页按帖子组织；媒体评论始终归媒体本体，而不是归某个帖子里的一份媒体副本。

---

### 3.4 系统媒体（System Media）
系统媒体指设备共享媒体库中的本地媒体映射结果。

系统媒体是**工具区能力**，不是 app 主内容流的一部分。

系统媒体用于：

- 浏览系统照片 / 视频
- 多选整理
- 发成帖子
- 加入已有帖子
- 移到系统回收站

系统媒体不支持：

- 评论
- 跳转已上传对应帖子
- 使用 app 内容区的原图加载策略

---

### 3.5 帖子评论（Post Comment）
帖子评论用于对整个帖子说话。

例如：

- 对这一组内容整体的感受
- 对一次旅行、一次见面、一个片段的补充

帖子评论只显示在帖子评论区。

---

### 3.6 媒体评论（Media Comment）
媒体评论用于对某一张图片 / 某一个视频说话。

例如：

- “这张拍得太好了”
- “这张你笑得很自然”
- “这段视频当时真的好吵哈哈”

媒体评论只显示在当前媒体对应的评论预览层和完整评论区中，不再进入帖子主评论区。

---

### 3.7 目录删（Directory Delete）
目录删是指：**仅从当前帖子中移除该媒体**。

它只影响当前帖子与媒体的关系，不删除媒体本体，不删除媒体评论。

---

### 3.8 系统删（System Delete）
系统删是指：**全局删除媒体本体**。

系统删会带来：

- 媒体从照片页媒体流消失
- 媒体从所有帖子中消失
- 媒体评论进入删除流程
- 原图 / 预览相关内容进入删除流程

在系统媒体工具区，系统删对应设备侧系统媒体删除 / 移到系统回收站；
在 app 内容区，系统删对应 app 内的全局媒体删除。

---

### 3.9 回收站条目（Trash Item）
回收站条目用于承载可恢复删除对象。

本版重点处理：

- 帖子删除
- 媒体目录删
- 媒体系统删

---

## 4. 信息架构与导航

### 4.1 全局底部导航
全局底栏固定为：

`主页 / 照片 / 生活（暂定）`

规则：

- 三项始终固定
- 不因照片模块内页面切换而变化
- 文件模块不再进入底栏规划

---

### 4.2 照片模块顶部二级导航
进入“照片”底栏后，顶部采用二级导航：

`照片 / 相册 / 回收站`

说明：

- **照片**：全局媒体流浏览入口
- **相册**：按相册浏览帖子
- **回收站**：删除恢复入口

---

### 4.3 照片模块顶部结构
为了让系统媒体入口更显眼、更方便，同时保持导航区足够紧凑，照片模块顶部采用**单行结构**：

#### 左侧：标题式二级导航
- 照片 / 相册 / 回收站
- 三项整体占据这一行左侧的主要空间
- 风格偏标题式导航，而不是厚重 tab

#### 右侧：工具入口区
- `系统媒体`
- 通知铃铛

规则：

- `系统媒体` 与通知铃铛在 `照片 / 相册 / 回收站` 三个二级页中都显示
- 二者位于同一水平线右侧，不再拆成上下两层
- `系统媒体` 仍采用带文字的轻按钮形式，方便识别和点击
- 工具区不应压过左侧标题式导航的章节感
- `照片 / 相册 / 回收站` 必须保持横向排布，不允许竖排或逐字换行
- 顶部区域应相对状态栏下沿更紧凑，避免空出过大的上边距，但仍需保留安全点击区
- 当前二级页标题需要更大、更突出；非当前标题更小、更弱，仍保持标题式导航而非厚重 tab
- `系统媒体` 在顶部工具区先压缩展示为 `系统`
- 通知入口使用铃铛图标按钮
- `照片 / 相册 / 回收站` 三个二级主页支持左右滑切换
- 左右滑切换只限二级主页，不适用于帖子详情页、Viewer、系统媒体等独立页面

---

### 4.4 页面骨架
照片模块统一采用：

- 顶部：固定导航区
- 中间：独立滚动内容区
- 底部：全局底栏固定

---

## 5. 照片页（全局媒体流）

### 5.1 定位
照片页是照片模块默认浏览入口。

其核心定位为：

- 展示 app 内容体系内的**全局媒体流**
- 以媒体为最小单位浏览
- 不在该页展示帖子卡片语义

---

### 5.2 数据范围
照片页展示的是：

- 已进入 app 内容体系的媒体
- 按媒体本体去重后的全局媒体集合

规则：

- 同一媒体即使属于多个帖子，在照片页中也只展示一次
- 一个媒体可以没有任何所属帖子
- 一个媒体可以属于一个或多个帖子

---

### 5.3 排序
默认按 `media_display_time` 从新到旧排序。

说明：

- 该时间用于照片页媒体流排序
- 允许后续在媒体编辑能力中修改
- 若媒体与帖子时间不同，照片页优先遵循媒体时间

---

### 5.4 网格密度
支持双指缩放，建议档位：

- 2 列
- 3 列
- 4 列
- 8 列
- 16 列

不保留 6 列。

在当前阶段，照片页应先具备基础可用的双指缩放切换列数能力；  
显式密度切换控件仍可保留，作为开发与验收阶段的补充入口。

---

### 5.5 分块层级与密度绑定
#### 当列数 = 2 或 3
- 按月分段加载
- 月内按日展示
- 显示日标题
- 日标题格式：`M月D日`

#### 当列数 = 4
- 按月分段加载
- 仍可显示日标题
- 以兼顾浏览效率与时间感知为主

#### 当列数 = 8
- 直接按月块展示
- 不显示日标题

#### 当列数 = 16
- 直接按年块展示
- 不显示日标题

说明：

- 分组逻辑要跟随密度切换即时变化
- 低密度优先保留更强的时间感知
- 高密度优先保留浏览效率与大时间块定位

---

### 5.6 卡片展示规则
照片页媒体卡片保持轻量，默认只展示：

- 媒体缩略图

不额外展示：

- 评论图标
- 评论数
- 所属帖子信息
- 归属感标记

原因：

- 保持全局媒体流纯净
- 关系和互动信息放到查看态再展示

---

### 5.7 交互
- 点击媒体：进入**媒体流查看态**
- 长按媒体：进入多选模式
- 滑动内容：可触发 Time Scrubber 浮现
- 点击系统媒体：进入系统媒体工具区
- 点击铃铛：进入通知中心

说明：

- `系统媒体` 与铃铛入口位于照片模块顶部右侧工具区
- 这两个入口在 `照片 / 相册 / 回收站` 三个二级页中保持可见
- 在早期实现阶段，可先交付轻量多选壳子：长按进入、基础选中状态、顶部切换为 `取消 + 已选 x 项` 的上下文栏
- 多选模式下的后续操作区不替换全局底部导航，而是在底部导航上方额外出现轻量上下文操作栏
- 点击媒体在当前阶段应进入照片页对应的 Viewer Shell v1，并基于当前全局媒体集合与当前点击位置打开对应媒体
- 当前阶段的 Viewer 先落地深色沉浸式骨架、左右滑切全局媒体流，以及左下评论入口 / 右下按钮组占位，不接真实评论、原图和所属帖子逻辑
- 后续正式返回规则中，系统返回应优先退出多选等当前状态，而不是直接退出 app

---

### 5.8 删除语义
从照片页媒体流发起删除时：

- 默认只有**系统删**
- 不提供“仅从某帖子移除”的入口
- 删除前进入全局删除确认流程

原因：

- 照片页是全局媒体维度
- 在这个视角下做删除，语义应直接指向媒体本体

---

## 6. 时间滑条（Time Scrubber）

### 6.1 目标
解决长时间流中间段定位困难的问题，核心是：

- 快速定位
- 分段加载
- 与网格层级一致

---

### 6.2 显示规则
- 默认隐藏
- 用户开始滚动后在右侧出现
- 停止交互后延迟淡出
- 视觉保持轻，不压过媒体网格本身
- 交互区域应覆盖右侧更完整的一整列纵向范围，而不是只停留在中间一小段

---

### 6.3 拖动态
按住拖动时：

- 滑条变粗，滑块变大
- 显示有内容的年份刻度
- 显示日期气泡，例如：`2026-02-19`
- 日期气泡跟随滑条当前位置一起移动，更像定位反馈而不是固定标签
- 内容区跟手滚动

---

### 6.4 定位规则
- **列数 <= 4**：优先定位到对应月份中的具体日分组
- **8 列**：优先定位到对应月份块
- **16 列**：优先定位到对应年份块

---

### 6.5 分段加载
- 月份作为一级加载节点
- 进入对应月份后增量拉取数据

---

## 7. 媒体查看态（统一规则）

### 7.1 设计目标
媒体查看态的目标是：

- 保持沉浸感
- 尽量不让厚重面板直接遮挡图片
- 让评论、原图、时间、所属帖子都可达
- 查看逻辑尽量统一

本版将“从照片页进入的媒体查看态”和“从帖子详情进入的帖子内查看态”做成高度统一的结构。

两者的主要差别只有：

- 从照片页进入时，左右滑切的是**全局媒体流**
- 从帖子进入时，左右滑切的是**同帖媒体**
- 帖子内查看态额外保留**分段小白条**

---

### 7.2 统一布局骨架
从上到下为：

1. 顶部控制区（返回）
2. 中部媒体画布
3. 画布上的沉浸式操作层
4. 底部位置指示区（仅帖子内查看态显示分段小白条）

说明：

- 这次不再把“时间胶囊 + 评论入口 + 原图按钮”统一做成图片下方固定信息行
- 查看态中改为**贴近画布边缘的沉浸式布局**
- 但仍需保证不遮挡主体、不喧宾夺主
- 媒体画布可整体略微上移，但必须与顶部返回区保持安全距离，并在视觉比例上始终大于评论预览层
- 顶部控制区保持极简，只保留返回标志，不显示来源提示与页码提示
- 系统状态栏区域应与查看态顶部统一为深色沉浸背景

---

### 7.3 画布上的沉浸式操作层
补充口径（以下内容覆盖本节旧描述）：

- 顶部结构改为三段式：左上仅保留 `<` 返回标志，顶部中间为轻量时间胶囊，右上为设置按钮占位
- 时间胶囊不再放在画布下方
- 顶部不显示来源提示与页码提示
- 系统状态栏区域需与查看态顶部统一为深色沉浸背景
- 媒体画布应尽量横向铺满，并以居中、自适应比例的方式呈现，不做卡片式留白
#### 左下角
放置：

- **评论气泡按钮**
- 评论数小标志

规则：

- 按钮固定位置，不可拖动
- 评论数小标志显示当前媒体评论数
- 有评论时正常显示数量；无评论时可只显示按钮，不显示数量

#### 画布下方
- 放置轻量**时间胶囊**

规则：

- 时间胶囊位于媒体画布下方，与画布保持小间距
- 需要避开左下评论预览层，不与其重合
- 视觉权重最轻，但仍可直接看到当前媒体时间

#### 右下角
建议采用竖向轻按钮组，依次放置：

1. `加载原图 / 原件`
2. `所属帖子`

规则：

- 两个按钮固定在画布右下区域
- 原图按钮与所属帖子按钮保持统一样式
- 若当前媒体不属于任何帖子，则隐藏“所属帖子”按钮或后续再改为“加入帖子”（非首版必做）
- 右下按钮区不承担顶部来源说明或页码提示

---

### 7.4 媒体评论的双层结构
补充口径（以下内容覆盖本节旧描述）：

- 评论预览层默认关闭，只保留评论气泡按钮
- 第一次点击评论气泡：打开评论预览层
- 评论预览层已展开后，第二次点击评论气泡：收起评论预览层
- 只有点击评论预览层中的评论条目时，才进入完整评论区，并可带占位定位能力
为保持沉浸性，媒体评论不再一上来就用完整 2/3 屏弹层遮住图片，而是采用**双层结构**。

#### 第一层：评论预览层
- 由左下角评论气泡按钮向上“投影”出一块**黑色半透明评论画布**
- 该评论画布固定位置，不可拖动
- 它属于查看态的一部分，不是独立弹窗

评论预览层的规则：

- 展示当前媒体的评论内容
- 每条评论默认展示 **2 行**
- 单行字数以后按实际 UI 视觉再定，此版只要求布局整齐、可读、节奏舒适
- 预览层内部支持上下滚动查看评论
- 若无评论，可展示轻量占位文案，例如“还没有评论”
- 若评论较多，仍只在同一个预览层中滚动，不自动展开成大面板
- 预览层可以比首版壳子更舒展，但仍应明显小于媒体画布
- 预览层高度与位置允许小幅微调，但必须保持“媒体画布 > 完整评论区 > 评论预览层”的层级关系
- 预览层应尽量贴近评论气泡，形成从气泡向上投影 / 发射出来的感觉

#### 第二层：完整评论区
以下操作会打开完整评论区（占屏约 2/3 到 3/4）：

- 点击评论预览层中的任意评论条目

完整评论区规则：

- 从底部弹出，占屏约 2/3 到 3/4
- 展示当前媒体的完整评论列表
- 支持输入新评论
- 若是从某条预览评论点击进入，需要自动跳转并定位到该条评论
- 完整评论区高度应高于评论预览层，形成从预览到详情的层级升级，但不应重到吞掉查看态

这样形成：

- 默认先看图 + 轻评论预览
- 想深入看或想输入时，再进入完整评论区

---

### 7.5 从照片页进入的媒体流查看态
其上下文语义是：

- 我正在浏览**全局媒体流**

规则：

- 未缩放时，左右滑切换全局媒体流中的前后媒体
- 不显示同帖分段小白条
- `所属帖子` 按钮用于跳转帖子上下文
- 首个可运行版本应基于当前照片页媒体集合与当前点击项打开，不额外引入帖子内分段信息

所属帖子按钮规则：

- 若当前媒体只属于 **1 个帖子**：点击后直接跳转
- 若当前媒体属于 **多个帖子**：点击后弹出底部列表，让用户选择目标帖子
- 若当前媒体暂无所属帖子：隐藏该按钮

---

### 7.6 从帖子进入的帖子内查看态
其上下文语义是：

- 我正在查看**当前帖子中的媒体**

规则：

- 布局、评论预览层、右下角按钮组与媒体流查看态尽量一致
- 未缩放时，左右滑切换同帖媒体
- 底部保留分段小白条

分段小白条规则：

- 同一帖每张媒体对应一段短白条
- 当前媒体及之前段高亮，其余置灰
- 长按可展开媒体缩略图带
- 长按滑动松手可跳转目标媒体

`所属帖子` 按钮在帖子内查看态也保留：

- 若当前媒体只属于当前这一个帖子，点击后可高亮反馈或无动作
- 若当前媒体属于多个帖子，则点击后可列出其它帖子，便于跳转

---

### 7.7 缩放规则
#### 未缩放
- 媒体流查看态：左右滑切全局媒体流前后项
- 帖子内查看态：左右滑切同帖媒体

#### 已缩放
- 只允许平移查看细节
- 禁止左右切媒体
- 可隐藏评论预览层、右下角按钮组和分段小白条
- 缩回适配屏幕后恢复原交互

---

### 7.8 返回规则
- 若完整评论区 / 所属帖子选择层已打开：先关闭当前 Viewer 内部面板
- 若当前处于缩放状态：先恢复适配
- 否则：退出查看态，返回进入前的位置
- 当前阶段至少要保证系统返回优先退出当前 Viewer，而不是继续向上触发 app 退出
- 作为后续正式要求，系统返回应优先退出当前瞬时状态，例如查看态先回媒体流、多选模式先退出多选
- 全局 app 退出不应一滑就直接退出，后续目标是先给轻提示，再次返回才退出；本阶段仅记录要求，不要求完整实现

---

## 8. 帖子详情页（Post Detail）

### 8.1 定位
帖子详情页是 app 内容体系中最重要的上下文页之一。

它负责：

- 浏览当前帖子的媒体序列
- 查看帖子信息
- 承载帖子评论
- 从帖子上下文进入查看态、编辑和导出

---

### 8.2 页面结构
从上到下为：

1. 媒体区
2. 图片下方信息行
3. 信息白区
4. 帖子评论区

这部分继续沿用你原来更喜欢的帖子详情思路，不改成相册式大列表页。

补充口径：

- 帖子详情页属于独立页面上下文
- 进入帖子详情页后，不继续显示全局底部一级导航
- 进入帖子详情页后，不继续显示照片模块顶部二级导航与其右侧工具区

---

### 8.3 顶部操作
帖子详情页右上角保留：

- `导出 / 保存` 入口
- `Gear Edit`（进入编辑页）

说明：

- 本阶段不强调复杂分享链路，顶部主语义以导出 / 保存为主
- 是否继续保留分享箭头图标，可在 UI 设计阶段再定

---

### 8.4 媒体区
- 展示当前帖子中的媒体序列
- 默认按帖子内媒体顺序展示
- 左右滑：切换同帖媒体
- 单击：进入帖子内查看态

---

### 8.5 图片下方信息行
图片下方信息行固定包含：

- 左侧：当前媒体时间胶囊
- 右侧：
  - 当前媒体评论图标
  - 评论数小标志
  - `加载原图 / 原件`

规则：

- 评论图标与评论数小标志绑定为一组
- 评论数显示当前媒体评论数
- 图标不覆盖图片，始终位于图片下方
- 点击评论图标，可直接打开当前媒体完整评论区

说明：

- 这里仍保留下方信息行，是因为帖子详情页本身不是全屏沉浸式查看态
- 这里的信息行承担“当前媒体快速操作”作用

---

### 8.6 信息白区
展示：

- 作者 / 贡献者标识
- 标题 / 简介（可空）
- `post_display_time`
- 所属相册 Chips
- `加载全帖原图` 文本按钮

说明：

- 所属相册 Chips 点击无跳转
- 长按所属相册 Chips 可作为进入编辑或移动归档的快捷入口（可选）

---

### 8.7 加载全帖原图
帖子详情页支持 `加载全帖原图` 能力。

规则：

- 入口位于信息白区中，作为帖子级操作
- 作用范围：当前帖子下的全部媒体
- 已加载过原图的媒体不重复下载
- 加载中可显示整体进度或简化状态
- 失败的单个媒体不影响其他媒体继续加载
- 加载完成后，帖子内各媒体都进入“已加载原图”状态

说明：

- `加载全帖原图` 是帖子级能力
- `加载原图` 是当前媒体级能力
- 两者不冲突

---

### 8.8 帖子评论区
帖子评论区只显示：

- **帖子评论**

明确不显示：

- 媒体评论
- 媒体评论聚合流
- 带媒体标记的评论条目

原因：

- 保持帖子评论语义清晰
- 避免重新回到“媒体评论混入总评论区”的旧问题

## 9. 相册页（按相册浏览帖子）

### 9.1 定位
相册页用于按相册浏览帖子。

它不是默认浏览入口，而是：

- 内容归档入口
- 主题浏览入口
- 帖子管理入口

---

### 9.2 页面结构
- 顶部：相册切换区
- 主体：当前相册下的帖子列表 / 帖子网格
- 右上角：相册管理入口（齿轮）

---

### 9.3 相册切换
相册切换采用轻量 chips 样式，当前口径优先为两行展示。

规则：

- 默认优先展示常用相册
- 当前相册 chips 需优先呈现两行结构，保持整齐、轻量、可扫读
- 若相册数量继续增多，可在两行基础上允许轻量横向滚动或继续合理换行
- 支持进入相册管理页进行增删改排序（具体管理细节可后置）

---

### 9.4 不做子相册
本版明确：

- 不做子相册
- 相册下直接承载帖子
- 不再继续向下套目录层

---

### 9.5 帖子列表展示
当前相册下展示帖子集合。

帖子列表支持：

- 点击进入帖子详情页
- 长按进入多选模式
- 批量导出
- 批量移动到其他相册（可选）
- 批量删除（可选）

Stage 4.1 先落地最小可用结构：

- 当前相册下先采用 2 列轻帖子卡网格
- 相册页列数状态当前支持 `2 / 3 / 4`，并补基础双指缩放切换能力
- 帖子卡至少包含封面、标题、一行摘要、轻时间信息和右下角媒体数
- 点击帖子卡先进入帖子详情页占位路由，用于打通浏览链路，不在本轮展开完整帖子详情能力

Stage 4.2 起，点击帖子卡进入独立帖子详情页 v1，不再停留在普通占位块。

---

### 9.6 帖子排序
默认按 `post_display_time` 从新到旧排序。

可选排序能力：

- 按时间
- 按修改时间
- 按手动排序（非首版必做）

## 10. 评论体系

### 10.1 总原则
评论拆分为两套并行但不混流的体系：

1. **帖子评论**
2. **媒体评论**

二者可以复用同一套输入器和排版工具，但展示流必须分开。

---

### 10.2 共同规则
适用于：

- 帖子评论区
- 媒体评论完整评论区
- 查看态中的评论预览层（仅展示规则部分复用）

统一规则：

- 最新在上
- 默认加载 10 条
- 第 10 条后显示“展开更多评论”
- 展开后支持下滑分页加载
- 支持“收起到最新 10 条”
- 单条长评论可折叠 / 展开

---

### 10.3 帖子评论
帖子评论的规则：

- 仅归属于帖子
- 仅显示在帖子评论区
- 不带媒体标记
- 不带跳转按钮

---

### 10.4 媒体评论
媒体评论的规则：

- 仅归属于媒体本体
- 仅显示在当前媒体的评论预览层与完整评论区
- 不进入帖子主评论区
- 若同一媒体属于多个帖子，则媒体评论仍只有一套

---

### 10.5 媒体评论入口
媒体评论入口存在于：

- 帖子详情页的当前媒体信息行
- 媒体流查看态左下角评论气泡
- 帖子内查看态左下角评论气泡

图标统一风格：

- 评论气泡按钮
- 评论数小标志

---

### 10.6 完整评论区
完整评论区用于：

- 查看当前媒体全部评论
- 输入新评论
- 编辑 / 删除 / 选择已有评论

规则：

- 从底部弹出，占屏约 2/3
- 打开后优先定位到触发位置相关的评论（若有）
- 关闭后回到原查看态，不打断当前媒体位置

---

### 10.7 单条操作
长按评论 Entry 后，弹出一级操作菜单：

- 编辑
- 删除
- 选择

规则：

- 编辑后不显示“已编辑”标志
- 删除不做恢复历史展示
- 点击“选择”后，进入文本选择状态
- 只有存在有效选区时，才出现“复制”操作
- 点击“复制”后，仅复制当前选中文本

---

### 10.8 回复功能
本版明确：

- **去掉评论回复功能**
- 不做引用回复
- 不做楼中楼
- 不做 `@` 提及逻辑

评论统一保持为普通平铺评论流。

---

### 10.9 输入器排版工具
可保留轻量快捷排版能力：

- 加粗（B）
- 项目符号（•）
- 数字序号（1.）
- 圆圈序号（①）

是否首版全部开放，可在实现阶段按节奏裁剪。

---

### 10.10 生命周期规则
#### 删除媒体时
- 若执行的是**目录删**：
  - 媒体本体仍存在
  - 媒体评论仍保留
  - 若该媒体仍可从照片页或其他帖子访问，则评论仍可查看

- 若执行的是**系统删**：
  - 媒体评论一起进入删除流程
  - 从所有评论入口中移除

#### 恢复媒体时
- 若媒体从系统删中恢复，则媒体评论一并恢复

#### 删除帖子时
- 帖子评论进入删除流程
- 若只是删除帖子本身，不自动删除仍存在的媒体评论
- 若删除帖子时同时选择系统删其中媒体，则被系统删的媒体评论也一起进入删除流程

---

## 11. 原图加载规则

### 11.1 适用范围
app 内容区内的媒体采用“预览图 + 原图”策略。

适用页面：

- 帖子详情页
- 媒体流查看态
- 帖子内查看态

不适用页面：

- 系统媒体工具区查看态

---

### 11.2 默认策略
- 默认显示 Detail Preview
- 不默认预取 Original

---

### 11.3 单媒体按钮状态
- 未加载：`加载原图`
- 加载中：`加载中...`
- 已加载：`已加载原图`

---

### 11.4 作用范围
单媒体 `加载原图` 只作用于当前媒体本体。

若同一媒体被多个帖子引用，则其原图加载状态可复用。

---

### 11.5 加载全帖原图
帖子详情页支持 `加载全帖原图`。

规则：

- 对当前帖子下所有媒体依次或并发拉取原图
- 已缓存原图的媒体直接跳过
- 单个失败不阻塞全帖流程

---

### 11.6 连续加载提示
当用户短时间内连续加载多张原图时，可提示：

`是否加载全帖原图？`

---

### 11.7 网络与缓存
- Original 不参与默认预取
- 原图下载进入 app 临时缓存 / 持久缓存策略
- 失败时保持 Detail Preview，不影响继续浏览
- 设置页应提供清理原图缓存与预览缓存的入口

## 12. 长图查看规则

### 12.1 适用范围
当图片纵横比满足建议阈值 `height / width >= 2.2` 时，按长图处理。

### 12.2 帖子详情普通态

- 横向切媒体需要更明显的横向手势

### 12.5 切换原图后的定位
- 切换到原图后尽量保持当前浏览位置

### 12.6 性能要求
- 禁止整张原始超长图一次性全尺寸解码

  - `取消`

- 设置完成后退出该模式

- 删除整个帖子
- 取消本次删除

## 18. 技术栈建议

### 1. 总体技术路线

本项目最适合采用 **Android 原生 + Kotlin + Jetpack Compose + Spring Boot** 的路线，而不是 Flutter、React Native、KMP 或前后端都走 Web 化方案。原因不是“原生更高级”，而是这个项目的核心能力高度依赖 Android 原生媒体体系：包括系统媒体读取、Photo Picker、MediaStore、沉浸式 Viewer、评论预览层、通知与权限引导，这些都和 Android 平台能力强绑定；同时 UI 设计稿已经明确把页面分成“照片页轻浏览、帖子详情稳展开、Viewer 沉浸式、系统媒体工具区”四种不同温度，Compose + Material 3 + 自定义设计系统最适合把这种差异做出来。  

整体架构建议采用 **MVVM + UDF（单向数据流）+ Repository**。也就是：UI 层只负责展示状态与派发事件；ViewModel 负责状态生产；Repository 负责本地数据、网络接口、系统媒体、缓存、上传下载等数据源整合。这样最适合你这种页面复杂、状态很多、后面还要不断改 PRD 的项目。

------

### 2. Android 客户端基础栈

**语言与 UI 层：**

- Kotlin
- Jetpack Compose
- Material 3
- 自定义 Design Token（颜色、圆角、字重、间距、阴影、深色 Viewer 风格）
- Navigation Compose 负责“底部一级导航 + 照片模块顶部二级导航”

**状态与依赖注入：**

- ViewModel + StateFlow
- Kotlin Coroutines + Flow
- Hilt 负责依赖注入，包括 Repository、UseCase、网络层、数据库层、WorkManager 依赖注入等

这里不建议再额外上很重的 MVI 框架，也不建议为了“架构更高级”再套一层复杂的 domain framework。对你这个项目来说，**Compose + ViewModel + StateFlow + Hilt** 已经是最稳、最适合长期维护的一套。

------

### 3. 照片页、帖子详情页、Viewer 的技术栈

这一块不建议依赖通用相册大组件库直接拼装，而应该采用 **“Compose 自研 Viewer 组件 + 专用图片/视频库”** 的方式。因为你的查看态不是普通图片预览，而是带有：沉浸式边缘操作层、评论气泡、评论预览层、完整评论区、帖子跳转、加载原图、帖子内分段小白条等强定制交互。通用图库通常能解决“看图”，但很难优雅解决你这套产品层逻辑。  

**图片加载建议：**

- 普通图片、缩略图、预览图：**Coil**
- GIF：Coil 的 GIF 扩展
- 超大图 / 长图：单独做“区域解码 / 分片解码”能力，不要只靠普通 `AsyncImage`
- 缓存：利用 Coil 的内存缓存 + 磁盘缓存，再结合你自己的原图状态管理

**视频播放建议：**

- **AndroidX Media3 / ExoPlayer**
- 短视频、帖子内视频预览、全屏查看都用同一套播放器基础能力，避免后面重复造轮子

**分页与长列表：**

- 照片页全局媒体流
- 相册页帖子列表
- 帖子评论列表
- 媒体评论列表

这些都建议接入 **Paging 3**。因为你的媒体流、帖子流、评论流本质都是“持续增长的大列表”，不适合一口气全量拉取。

------

### 4. 系统媒体工具区的技术栈

系统媒体工具区建议单独看待，它不是你 app 内容体系的一部分，而是一个 **基于 Android 原生媒体体系的工具型区域**。这一块的核心技术是：

- **MediaStore**：读取设备里的图片 / 视频
- **Photo Picker**：作为补充选图入口，以及权限更友好的兜底方案
- 适配 Android 14 的 **Selected Photos Access（部分照片授权）**
- 你自己 app 创建的媒体，在 Android 10+ 上不需要额外存储权限即可访问；但读别的 app 创建的媒体，仍要按系统权限模型来处理

这一块的实现建议是：

- 网格展示走 Compose + MediaStore 查询结果
- 列表缩略图仍由 Coil 负责
- 单张查看走“系统媒体查看态”，不接入评论、不接入 app 内原图策略、不接入帖子跳转
- 删除统一走系统回收站语义，单张和多张都只弹一次系统确认

也就是说，**系统媒体区是“工具区”，不是“内容区”**，技术上和照片页共享部分列表基础能力，但业务逻辑要彻底分开。 

------

### 5. 本地数据层

本地数据层建议分成两层：

#### 5.1 Room：存结构化业务数据

用于存：

- 相册
- 帖子
- 媒体元数据
- 帖子评论缓存
- 媒体评论缓存
- 回收站条目
- 上传 / 下载任务状态
- 通知已读状态
- 系统媒体的轻量映射状态（如“已发帖”标记、最近同步时间等）

Room 是 Android 官方推荐的 SQLite 抽象层，适合这种结构化、关系明确、需要离线浏览和本地缓存的业务。

#### 5.2 DataStore：存轻量设置项

用于存：

- 照片页网格密度
- 上次停留页面
- 主题与 Viewer 偏好
- 通知开关
- 是否已看过权限引导
- “首次默认帖子流 / 以后记住上次选择”这类轻配置

DataStore 适合保存异步、事务性、轻量键值或类型化对象配置，应该替代 SharedPreferences。

简单说就是：
 **Room 管业务数据，DataStore 管设置状态。**

------

### 6. 网络层与接口风格

客户端网络层建议采用：

- **Retrofit**：定义 REST 接口
- **OkHttp**：底层 HTTP 客户端
- **kotlinx.serialization**：JSON 序列化 / 反序列化

其中：

- Retrofit 负责把接口定义写成 Kotlin 接口，最适合你现在这种“接口多、结构清楚、以后还要频繁改字段”的项目
- OkHttp 负责超时、重试、拦截器、认证头、日志等底层能力
- `kotlinx.serialization` 负责数据模型和 JSON 对齐，和 Kotlin 配合最好

接口风格建议优先采用 **REST**，不建议首版上 GraphQL。你的领域对象（相册、帖子、媒体、评论、回收站、通知）边界很清晰，REST 足够用，而且更容易和 Retrofit、分页、本地缓存一起配合。

------

### 7. 上传、下载与同步

这部分是你项目的核心之一，建议分成三层：

#### 7.1 客户端任务调度：WorkManager

所有“必须稳定完成，但不要求马上在前台盯着跑”的工作，都交给 WorkManager，例如：

- 媒体上传
- 原图后台下载
- 失败重试
- 帖子草稿补提交
- 系统媒体状态同步
- 回收站延迟清理后的后台请求

#### 7.2 媒体上传：直传对象存储

对你这个以图片 / 视频为核心的项目，最适合的上传架构不是“所有文件都先绕后台服务器”，而是：

1. app 先向业务后端申请临时上传凭证
2. 后端只负责签发临时凭证 / 记录业务元数据
3. app 直接把文件上传到对象存储
4. 上传成功后再回写帖子、媒体、封面、评论关联等业务信息

如果你用阿里云，官方推荐的移动端直传方案就是 **STS 临时凭证 + App 直传 OSS**，而不是把永久密钥写进 app，也不是把大文件全部先穿过业务服务器。大文件上传还可以直接用 OSS 的分片 / 断点续传能力。

#### 7.3 下载策略

- 列表：缩略图
- 帖子详情：预览图优先
- Viewer：按需加载原图
- 视频：流式播放优先，必要时缓存
- 系统媒体区：直接本地读，不走云端原图策略

这套策略和你 PRD 里“照片页轻、Viewer 沉浸、系统媒体工具化”的分工是完全一致的。 

------

### 8. 通知与消息

首版通知建议采用：

- **FCM（Firebase Cloud Messaging）** 做推送
- Android 通知通道（Notification Channels）
- Android 13+ 的 `POST_NOTIFICATIONS` 运行时权限

原因很简单：
 你这是两人长期使用的照片产品，不是强 IM 工具。首版最需要的是“评论了你”“删了你们共同内容”“上传完成了”这类 **可靠推送 + 进入 app 后刷新状态**，而不是一上来就做真正的实时聊天系统。

所以我建议：

- **首版不上 WebSocket**
- 评论、删除、恢复、帖子更新，用 FCM + app 内刷新就够了
- 真要做强实时聊天，再单独加 Spring WebSocket

Spring 官方确实提供了完整 WebSocket 支持，但它更适合你以后真的要做聊天、实时输入状态、多人协作时再上。

------

### 9. 服务端技术栈

服务端最适合你的，不是“越新越好”，而是“**稳、成熟、你能掌控**”。

#### 9.1 基础框架

- **Spring Boot 3.5.x**
- **Java 17**
- Maven 或 Gradle 二选一，优先 Maven（对你更省心）

Spring Boot 官方当前同时有 3.5.x 和 4.0.x 稳定线；对你这种首版要稳、要快落地、又以 Spring Web / Security / JPA 为主的项目，更建议先用 **3.5.x**，不要急着追 4.0.x。

#### 9.2 Web 与业务层

- **spring-boot-starter-web**
- 业务按模块拆包，不要一上来拆微服务
- 接口全部走 REST
- 模块建议按：认证、相册、帖子、媒体、评论、通知、回收站、系统媒体导入、上传凭证 这几个方向拆分

#### 9.3 安全与鉴权

- **Spring Security**
- 建议做 **Access Token + Refresh Token** 的移动端鉴权方案
- 图片 / 视频的上传下载权限、评论权限、删除权限都统一在服务端做校验

#### 9.4 数据访问

- **Spring Data JPA**
- 数据库建议 **PostgreSQL**
- JPA 足够支撑你首版的相册、帖子、媒体、评论、通知、回收站这些关系型数据

------

### 10. 媒体存储与云资源

媒体文件不要落数据库，也不要长期堆在业务服务器本地磁盘。最适合的是：

- **媒体二进制文件：对象存储（阿里云 OSS）**
- **业务元数据：PostgreSQL**
- **临时缓存 / 缩略图状态 / 任务状态：客户端 + 服务端各自做轻缓存**

阿里云 OSS 本身就是为海量对象存储设计的，并且官方已经提供了 Android 端上传、下载、断点续传、移动端直传等能力。对你的项目来说，这比“自己维护文件服务器 + Nginx 静态盘”更稳，也更适合图片 / 视频业务。

------

### 11. Redis、WebSocket、实时能力：首版怎么取舍

这三个能力都不是“不能用”，而是 **首版不要默认全上**。

#### 11.1 Redis

建议作为 **二阶段可选** 能力，用于：

- 通知去重
- 热点缓存
- 短期会话 / 验证码
- 限流
- 异步任务状态共享

Spring Data Redis 很成熟，但你的首版不是高并发平台，没必要一开始就把复杂度拉高。

#### 11.2 WebSocket

建议后置。只有当你后面真的要做：

- 实时聊天
- 实时输入状态
- 在线协作提醒
- 强实时通知流

再加 Spring WebSocket。首版“评论通知 + app 内刷新”不需要它。

------

### 12. 观测、日志与运维

虽然你项目一开始用户很少，但这块建议从第一版就打基础，不然后面很难补。

服务端建议直接带上：

- **Spring Boot Actuator**
- **Micrometer**
- 至少保留 `health`、`metrics`、基础日志
- 后续如有需要再接入 tracing

客户端建议至少做好：

- Crash 收集
- 上传失败埋点
- Viewer 原图加载失败埋点
- 系统媒体导入失败埋点

你这个项目的难点不是“登录接口能不能通”，而是媒体、评论、删除、回收站这些链路一长就容易出问题，所以可观测性一定要从第一版就留接口。

------

### 13. 首版明确不建议使用的技术

首版不建议使用：

- Flutter / React Native / WebView 壳
- Kotlin Multiplatform
- 微服务拆分
- GraphQL
- 自建文件模块
- 首版强依赖 Redis
- 首版强依赖 WebSocket
- 重型富文本编辑器
- 通用大而全相册组件库

原因不是这些技术不好，而是它们 **不适合你现在这个阶段**。你现在最重要的是：把“照片、帖子、评论、Viewer、系统媒体、回收站”这条主链做稳、做顺、做漂亮。

### 14. 最终推荐栈（简版结论）

**客户端：**

- Kotlin
- Jetpack Compose + Material 3 + 自定义设计系统
- Navigation Compose
- ViewModel + StateFlow
- Hilt
- Coroutines + Flow
- Coil
- Media3 / ExoPlayer
- Room
- DataStore
- Paging 3
- Retrofit + OkHttp + kotlinx.serialization
- WorkManager
- MediaStore + Photo Picker

**服务端：**

- Java 17
- Spring Boot 3.5.x
- Spring Web
- Spring Security
- Spring Data JPA
- PostgreSQL
- 阿里云 OSS
- FCM
- Spring Boot Actuator + Micrometer

**二阶段可选：**

- Redis
- Spring WebSocket

**总体原则：**

- 客户端原生优先
- Viewer 与系统媒体分别实现，不混逻辑
- 媒体文件直传对象存储
- 元数据与业务关系走 PostgreSQL
- 首版稳、轻、少依赖，二阶段再补实时能力与缓存能力
